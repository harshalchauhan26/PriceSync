# MBO Tracker ‚Äî Audit & Findings (2026-07-21)

Read-only review of the live Node/Express + React + Supabase stack under `web/`. This is a mature, well-defended codebase ‚Äî most classic mistakes are already handled (parameterized SQL, encrypted Shopify token, graceful shutdown, advisory-lock migrations, cents-descale guards). The findings below are the real gaps that matter for the four asks: verify the price-fetch links, retry broken links before voiding, harden for production, and close security holes.

Severity key: **P0** = do before any bulk/live action ¬∑ **P1** = fix soon ¬∑ **P2** = improve.

## 1. Price-fetch links & price correctness

**How it works today.** `startPipeline` (pipeline.js) pulls work rows from Supabase, fans them across worker threads (`worker.js` ‚Üí `engine.extractRow`), then `finalizeOne` compares the fetched price to the baseline and writes state `matched` / `mismatch` / `error`. Broken links become `Fetch Error (...)` rows in state `error`.

**Retry-before-void already exists.** When `safe_retry` is on, the pipeline reruns every `error` row a second time gently (concurrency 1, 4‚Äì8s cooldowns, 15s timeout) ‚Äî `startPipeline` lines 375‚Äì389. Individual rows can also be re-fetched from the Review ‚Üí Errors tab (`/api/review/rerun` ‚Üí `pipe.rerunOne`). **Nothing is ever auto-deleted**; error rows persist until a human acts. So the "recheck before deeming void" behavior is present ‚Äî but it's a single extra pass gated on a toggle, and there is no report that tells you *which* errors are transient (timeout/429/5xx) vs. permanent (404 / removed product).

**Price-correctness risk (the crux of "recheck if price fetched is correct").**
- Cents-descaling is heuristic: `descaleIfCents` divides by 100 above a 1,000,000 threshold, and only for `source === "json"` (engine.js 156‚Äì158, 304‚Äì305). The code comments record a live incident where a genuine ‚Çπ1.2M price was halved. The current guard is good but threshold logic is inherently fragile at the extremes.
- Currency detection (`detectCurrency`) falls back to `$ ‚áí USD` last; a geo-mislabeled store can report the wrong currency. This is patched per-brand via `native_currency_brands` / `usdFetchBrands`, i.e. by hand.
- `redirectedOffProduct` guards against removed products that 302 to a category page, but only if a final-URL is observable (`responseUrl` or the relay's `x-relay-final-url`); some proxy responses expose neither and would slip through.
- `sanitizePrice` strips commas, spaces and apostrophes, so a European decimal-comma price (`1.234,56`) would parse wrong ‚Äî low impact since only USD/CAD/INR are in scope.

**Gap:** there is **no reconciliation/verification report**. "Is the fetched price correct?" is currently answered only by the human eyeball in Review. The Claude Code prompt below builds a read-only verification script that flags the tell-tale bad patterns (‚âà100√ó, ‚âà¬Ω, ‚âà0.01√ó ratios; currency ‚âÝ expected; error clustered by brand).

**Gap:** no automated test suite exists (only `simulation` mode and `_*.mjs` scratch probes). Core pure functions (`sanitizePrice`, `descaleIfCents`, `detectCurrency`, `redirectedOffProduct`, `roundFinal`, `computeFinal`) are ideal unit-test targets and currently untested.

## 2. Production readiness

**Strong already:** `wrap()` around every handler, `unhandledRejection`/`uncaughtException` logging, SIGTERM/SIGINT graceful drain, `EADDRINUSE` handling, health check at `/api/health`, non-blocking advisory-lock schema migration with retry, upsert-only imports that can't wipe the catalog.

- **P1 ‚Äî Single-instance only, by design.** Sessions, login rate-limits, pipeline engines, the push chain and push jobs all live in process memory (`security.js` `SESSIONS`/`FAILS`, `pipeline.js` `ENGINES`, `shopify.js` `_pushChain`, `push-job.js` `jobs`). Running 2+ instances silently breaks auth, rate-limiting and "only one push at a time." PRODUCTION.md says run one ‚Äî keep it that way, or move this state to Postgres/Redis before scaling.
- **P1 ‚Äî 500 responses leak `e.message`** to the client (`wrap`, server.js 47‚Äì49). Fine internally; log server-side and return a generic message.
- **P2 ‚Äî Upload limits inconsistent.** `multer` is hardcoded to 64 MB; the `MAX_UPLOAD_MB` env var is never wired in. `express.json` is 2 MB. Wire the env and lower the multer cap to something realistic for the sheet sizes you use (~7 MB).
- **P2 ‚Äî Heavy full runs share the web process.** Worker threads for a 6k-product run compete with request serving on a small instance. Acceptable per the current design; just don't kick off a full run during heavy user traffic.
- **P2 ‚Äî No structured request logging / request IDs**, which makes production incident triage harder.

## 3. Security

- **P0 ‚Äî Open self-registration exposes all business data.** `/api/register` is public and creates a `viewer` (server.js 60‚Äì67). `guard` only blocks *writes* for non-admins ‚Äî every authenticated user, including a self-registered viewer, can `GET` the entire catalog, pricing, insights, alerts, history and active sessions-adjacent data. Google sign-in (server.js 70‚Äì87) has the same effect for any verified Google email. **Anyone who can reach the URL can read everything.** Restrict registration to an allowlist/invite, or disable it and create users from the owner console.
- **P0 ‚Äî Weak admin password, and it can't be rotated via env.** Config default is `admin`; PRODUCTION.md records the dev value `harshal123`. `seedOwner` (security.js 73‚Äì80) only creates or promotes ‚Äî it never updates an existing owner's password, so changing `ADMIN_PASSWORD` does nothing once the owner exists. Rotate in-app (or via a one-off DB update) to a strong password.
- **P0 ‚Äî Shopify push is LIVE (`dry_run=0`).** `_pushPrice` writes real Shopify prices unless `dry_run` is set (shopify.js 141‚Äì145). Any test run that ends in an approve+push will change live storefront prices. Set dry-run before testing.
- **P1 ‚Äî Login rate-limit is bypassable via `X-Forwarded-For` spoofing.** `ipOf` (security.js 83‚Äì86) trusts the left-most XFF value, which the client controls; rotating it defeats the 5-fails/10-min lock. With `trust proxy` = 1 on Render, prefer `req.ip` (or the right-most XFF hop) as the limiter key.
- **P2 ‚Äî No CSRF tokens.** Mitigated in practice by `sameSite: lax` cookies plus JSON-only body parsing (a cross-site HTML form can't send `application/json`), so risk is low for this internal tool ‚Äî but worth a note if any endpoint ever accepts form encoding.
- **Good, confirmed:** `.env` is gitignored and untracked; SQL is parameterized throughout (dynamic brand filters still use `$n`, state filters use whitelist maps); Shopify token encrypted at rest (AES-256-GCM); security headers set and `x-powered-by` disabled; no CORS (same-origin only); Shopify calls have cost-aware throttle/back-off.
